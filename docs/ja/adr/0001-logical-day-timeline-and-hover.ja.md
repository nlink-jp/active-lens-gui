# ADR 0001 — 論理日タイムラインの描画とセグメントのホバー

**ステータス:** 提案（合意待ち）
**日付:** 2026-07-10
**対象:** `AnalysisView`、`Models`、`ActivityModel`
**対になる ADR:** `active-lens` ADR 0001（セッション基準の稼働日帰属）— 両者は同時に
リリースする。本アプリは新ペイロードなしに描画できず、エンジン側の変更は本変更なしに
可視化されない。

---

## 1. 背景

3 つの変更が同時に入る。理由はそれぞれ別にある。

**エンジンの「日」はもうカレンダー日ではない。** `active-lens` ADR 0001 は深夜 0 時での
バケット分割を論理日（`[D 04:00, D+1 04:00)`）へ置き換え、稼働セッションを丸ごと
開始日へ帰属させる。したがって、ある日のセグメントは日の原点から 24 時間を超えて
伸びうる。00:59 まで続く夜のセッションは、04:00 に始まった論理日の 16.7 時から
20.98 時にあたる。現在のチャートはカレンダー深夜 0 時からの時間を測り、y 範囲を
`min(24, …)` でクランプしているので、この修正が保とうとしているスパンを
正確に切り落としてしまう。

**セグメントへのホバーは物理的に成立しない。** 要望されたポップオーバーは
スパンの Start / End / Duration を出す。しかしタイムラインが描くのは生のセグメントで、
生のセグメントは極小である。記録された 2026-07-09 のデータでは `operating` と
`present` が数分ごとに交替している（`20:53–20:55 present`、`20:55–20:56 operating`、…）。
チャートの高さは約 13 時間に対して 260 pt、つまり **1 分がおよそ 0.33 pt**。
2 分未満のセグメントはサブピクセルであり、カーソル直下のセグメントは
隣接するものの中から事実上ランダムに選ばれる。

**メニューバーの単位は日ではなくセッションである。** §2.3 を参照。

## 2. 決定

### 2.1 論理日の原点から時間を測る

`TimelineDay.hours(_:)` を `Double(unix - dayStartUnix) / 3600.0` にし、
`Format.midnightEpoch(date)` ではなく新しい `day_start_unix` を使う。24 を超える値は
正当な値であり、クランプしない。

`hoursBounds` から `min(24, …)` の上限を外す。下限は 0 のまま（論理日が自身の原点より
前に始まることはない）。y 軸の目盛ラベルはオフセットを `(day_start_hour + h) mod 24` で
実時刻へ戻す。未明まで続くセッションは列を下るにつれ `22:00`、`0:00`、`2:00` と読める —
軸は実際の時計の時刻を示したままで、列だけが 1 日より長くなることを許される。

### 2.2 ホバーはセグメントではなくブロックを報告する

エンジンは日ごとに `blocks[]` を出す。`work` ブロックは active セグメントの連続、
`break` ブロックは `break_minutes` 以上の away スパン。両者が各セッションを敷き詰め、
1 日あたり数百ではなく数個になる。ブロックは目に「1 本のバー」として映るものであり、
指し示せる大きさがある。

`.chartOverlay` と `onContinuousHover` で、`ChartProxy.value(atX:as:)` /
`value(atY:as:)` を介してカーソルを `(date, hourOffset)` へ写像し、純粋関数
`blockAt(day:hourOffset:)` がそのオフセットを含むブロックを見つける。
ツールチップのカードが出すのは以下。

- `work` ブロック — **Start**、**End**、**Duration**、その下に operating / present の内訳。
- `break` ブロック — **Start**、**End**、**Duration**、休憩である旨のラベル。

カーソルが日の列の上にあってもどのブロックにも乗っていなければ、カードは出さない。
カードはカーソルの脇に置き、チャート境界内へクランプするので、右端や下端で画面外に
描かれることはない。チャートから出るとカードは消える。

ブロックの算出は CLI 側で行い、ここでは行わない。`CLAUDE.md` は導出の所有者を
エンジンと定めており、純粋な Go 関数は Swift の関数よりテストが安い。

### 2.3 メニューバーは「today」ではなく now session を示す

メニューバーの単位は、そもそも日ではなかった。`PopoverView.workSession(_ d:
TimelineDay)` はセッションの名を冠し、`started …` / `ended …` を出しながら、
day を受け取っている。本変更の前はカレンダー日がちょうど 1 つの稼働セッションの
外枠を導出していたので、この抽象の漏れが表面化しなかっただけである。セッションを
一級市民に昇格させた今、この代理は不要であり、01:00 や徹夜明けでは端的に誤りになる。

ポップオーバーとメニューバーのタイトルは `active-lens now --json` を読む。これは
now session（open / paused、開始、終了、active の内訳、休憩）を、それが帰属する
論理日とともに返す。`ActivityModel.todayTimeline` は `nowSession` に、
`todayActiveLabel` は `sessionActiveLabel` になる。

- **メニューバーの見出し** — now session の active 時間。「この一連の作業がどれだけ
  続いているか」に答える。一瞥するとはそういうことである。
- **ポップオーバー本体** — セッション: 開始時刻、active、operating / present の内訳、休憩。
  paused のときは黙って止まるのではなく、その旨を表示する。
- **ポップオーバーの二次行** — `Today`、論理日の active 合計。同じペイロードの
  `day` フィールドから取る。

`todayString()` と `calendarSince(_:)` は削除する。どちらもエンジンが所有するように
なった日付演算を再実装したものだ。前者は論理日の下で誤ったキーを引き、後者は範囲を
カレンダー上の今日に固定してしまう。分析ウィンドウは `timeline --days N --json` を
呼び、範囲の解決は CLI に任せる。

## 3. 帰結

日の列は 24 時間より高くなりうるので、共有された y スケールは範囲内で最も長い日に
合わせて伸びる。エンジン側 ADR が述べる病的ケース（2 日間ディスプレイが眠らず
約 48 時間のセッションが生じる）では、チャート全体の y 範囲がそれに合わせて伸びる。
自動フィットは正しく、ただ結果が縦に長いだけである。

夜間の就寝は away バーとして現れなくなる。エンジンがセッション外のセグメントを
出さなくなるためだ。凡例の `Away` は、稼働セッション内の短い離席と休憩のみを
指すようになる。これは稼働ログが示すべきものそのものである。

メニューバーの数字は新しいセッションが開くたび `0s` にリセットされる。一晩眠った翌朝
07:26 に Mac を起こしても、深夜前に働いた時間はもう出ない。これはセッションを見出しに
据えたことの意図された意味だが、この 2 本の ADR の中で最も可視的な挙動変更である。
従来の数字をクリック 1 回で取り戻せるように `Today` 行を置く。

## 4. 却下した代替案

**それでも生セグメントをホバーする。** 元の要望どおりであり、参照は 1 行で済む。
しかし 1 分 0.33 pt では答えが予測不能になる。ユーザーはバーを指したつもりで、
カーソル下にある 4 つのサブピクセルセグメントのうちたまたま勝ったものを受け取る。
ポインタが 1 ピクセル動くたびに変わるツールチップは、ユーザーに何も教えない。

**2 段カード（上にセグメント、下にブロック）。** 情報量は最大で、両方を見せることで
曖昧さも解ける。却下の理由は、セグメント行が予測不能さを抱えたままで、しかも
対照される安定した行が隣に並ぶこと、そしてカードが大きくなり、検分している当の列を
覆い隠すこと。

**論理日の合計をメニューバーの見出しのまま残す。** 数字がリセットされないのは心地よい。
却下の理由は、それがメニューバーに対して誰も投げない問いに答えているからだ。09:00 に
始まった日の 20:00 に「今日 9h 40m」と言われても、いま自分がいる作業の連なりについては
何もわからないし、徹夜明けにはその数字は何時間も前に終わった日のものである。合計は
ポップオーバーの 1 行の価値はあるが、見出しの価値はない。

**日跨ぎセッションを翌日の列へこぼす。** どの列もちょうど 24 時間に保てる。しかし
エンジンの変更が取り除こうとしている深夜 0 時の視覚的断絶を再導入してしまい、
稼働ログの行は 1 つのセッションのために 2 つの列を挙げる羽目になる。

## 5. テスト計画

- `TimelineDay.hours(_:)` が `day_start_unix` からのオフセットを返す。翌日の原点を
  越えるセッションでは 24 を超える値も含む。
- 軸ラベルの写像: `day_start_hour = 4` でオフセット 20 は `0:00` と描かれる。
- `blockAt(day:hourOffset:)` が包含ブロックを返し、空白では `nil`、最後のブロックより
  後ろでも `nil` を返す。
- `blocks`、`sessions`、`day_start_unix` を含むペイロードのデコード。および null 配列に
  対する既存の防御的フォールバック。
- `now --json` のデコード: open なセッション、paused なセッション、closed なセッション、
  および null セッション。
- `sessionActiveLabel` がセッションの active 時間を出し、エラー時は `—`、セッションが
  まだ無いときは `0s` を出す。
- `Format.duration` が CLI の `formatSeconds` と歩調を合わせ続ける（既存テスト、変更なし）。
